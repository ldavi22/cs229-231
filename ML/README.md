# House Prices - Advanced Regression Techniques

## მიმოხილვა

მოცემული კონკურსი ეხება სახლის ფასების წინასწარმეტყველებას არსებული დატასეტის გამოყენებით, რომელიც შეიცავს 79 სხვადასხვა თვისებას. ჩვენი მიზანია სხვადასხვა თვისებების დამუშავების ტექნიკებითა და მოდელის ტუნინგით სასურველი შედეგის მიღება.

## ჩემი მიდგომა

ამოცანა უნდა დავყოთ რამდენიმე ეტაპად:

1. **მონაცემების კლასიფიკაცია** — უნდა გადავხედოთ არსებულ თვისებებს და აღვნიშნოთ თვისების ტიპები (რიცხვითი / კატეგორიული)
2. **Feature Engineering** — ვამუშავებთ კატეგორიულ ცვლადებს და გადაგვყავს სხვადასხვა ტექნიკით რიცხვებში. გარკვეული თვისებების შემთხვევაში, ვქმნით ახალ თვისებებს, რომლებიც ამცირებენ სვეტების რაოდენობას
3. **Feature Selection** — სხვადასხვა ფილტრის საშუალებით (RFE, Correlation) ვფილტრავთ თვისებებს
4. **მოდელების ტრენინგი** — სხვადასხვა წრფივ და ხის მოდელებს ვუკეთებთ ტრენინგს და მათვის შესაბამის ჰიპერპარამეტრების ოპტიმიზაციას
5. **ლოგირება და შედარება** — ყველა ექსპერიმენტი ილოგება MLFlow-ზე და ხდება მათი შედარება, შემდგომ კი საუკეთესოს არჩევა

## Feature Engineering

### კატეგორიული ცვლადების რიცხვითში გადაყვანა

#### ცვლადების ტიპების კლასიფიკაცია

პირველი ნაბიჯი იყო ცვლადების დაყოფა. მნიშვნელოვანი იყო კატეგორიული ცვლადების კლასიფიკაცია ქვეკატეგორიებად, რადგან ეს უფრო ცხადს ხდიდა რა ტიპის "ტექნიკა" უნდა გამოგვეყენებინა კონკრეტული ცვლადისადმი.

| ქვეკატეგორია | მაგალითი |
|---|---|
| 1) უწყვეტი | LotArea, GrLivArea |
| 2) დისკრეტული | Fireplaces, GarageCars |
| 3) Ordinal (კლასი შეიძლება იყოს უკეთესი მეორეზე) | OverallQual, ExterQual |
| 4) Nominal (ტიპს არ აქვს მეორე ტიპთან უპირატესობა) | Neighborhood, BldgType |
| 5) დროის აღმწერი | YearBuilt, YearRemodAdd |
| 6) Binary (მხოლოდ ორი შესაძლო მნიშვნელობა) | Street, CentralAir |

**რატომ დავყავით ასე?** `MSSubClass` არის იმის მაგალითი თუ რატომ გახდა საჭირო მსგავსი მიდგომა. ამ თვისებაში არსებული რიცხვები აღწერენ კონკრეტული სახლის ტიპს. ეს რიცხვები უბრალოდ უნიკალური კოდებია და არ ნიშნავს, რომ მაღალი რიცხვის მქონე კლასი ჯობია დაბალი რიცხვის მქონეს. მიუხედავად იმისა, რომ ეს ჩვენ ვიცით, წრფივი მოდელი ამას ვერ აღიქვამს და ჰგონია, რომ ერთი მეორეზე უკეთესია რიცხვითი მონაცემის მიხედვით. შესაბამისად, უფრო მეტი წონა ენიჭება მაღალი რიცხვის მქონე კლასს. ამ უკანასკნელის ასე დატოვება არ შეიძლება, ვინაიდან ხმაურს დაამატებს ჩვენს შედეგს, ამიტომაც მსგავსი ტიპის თვისებები საჭიროა, რომ გავაერთიანოთ Nominal ქვეკატეგორიის ქვეშ.

#### Ordinal კოდირება

Ordinal ცვლადების გარდასაქმნელად შეგვიძლია გამოვიყენოთ Ordinal Encoding, რომელიც რიცხვს უსაბამებს ტიპს მისი "ძლიერების" მიხედვით. ჩვენს შემთხვევაში ცვლადები, რომლებიც იზიარებენ Ex > Gd > TA > Fa > Po ხარისხის შკალას, ყველა ერთი და იგივე მიდგომით გარდაიქმნა: None=0, Po=1, Fa=2, TA=3, Gd=4, Ex=5.

ამ ტიპის თვისებებია: ExterQual, ExterCond, BsmtQual, BsmtCond, HeatingQC, KitchenQual, FireplaceQu, GarageCond, GarageQual.

სხვა Ordinal ცვლადებისთვის შეიქმნა ინდივიდუალური mapping-ები:

- **BsmtExposure:** None(0) → No(1) → Mn(2) → Av(3) → Gd(4)
- **BsmtFinType1/2:** None(0) → Unf(1) → LwQ(2) → Rec(3) → BLQ(4) → ALQ(5) → GLQ(6)
- **GarageFinish:** None(0) → Unf(1) → RFn(2) → Fin(3)
- **Functional:** Sal(1) → Sev(2) → Maj2(3) → Maj1(4) → Mod(5) → Min2(6) → Min1(7) → Typ(8)
- **Fence:** None(0) → MnWw(1) → GdWo(2) → MnPrv(3) → GdPrv(4)
- და სხვა: LotShape, LandSlope, PavedDrive, Utilities, Electrical

**მიდგომის ახსნა:** შეირჩა თანაბარი ინტერვალების მქონე მაპინგი (1, 2, 3...). ეს ოპტიმალურია ხეზე დაფუძნებული მოდელებისთვის, რადგან ხეები მხოლოდ threshold-ებს ეძებენ და მის მიხედვით იყოფიან, ინტერვალებს არ აქვს მათთვის მნიშვნელობა. თუმცა, წრფივი მოდელებისთვის ეს შესაძლოა არაოპტიმალური იყოს, მაგალითად: OverallQual-ში ხარისხი 8-დან 10-მდე ნახტომი შესაძლოა $80,000 ღირდეს, ხოლო 4-დან 6-მდე მხოლოდ $15,000. თანაბარი ინტერვალები ამ განსხვავებას ვერ ასახავს, მიუხედავად ამისა, დავალებაში მაინც თანაბარი ინტერვალების მქონე მეპინგი გვაქვს.

#### One-Hot კოდირება ნომინალური ცვლადებისთვის

ნომინალური ცვლადები, რომლებიც ზემოთ იქნა აღწერილი, გარდავქმენით `One-Hot` კოდირების საშუალებით. ეს ტექნიკა ყოველი არსებული ტიპისთვის თვისებაში ქმნის ბინარულ სვეტს, რომელშიც 1-იანი არის ის ყველა სტრიქონი, რომელსაც გააჩნია ამ თვისების შესაბამისი ტიპი, 0-იანი თუ პირიქით. ამ მიდგომის მინუსი არის ის რომ, თუ თვისებას გააჩნია N შესაძლო მნიშვნელობა, სვეტების რაოდენობა იზრდება N-ით, რაც ართულებს მუშაობის პროცესს და უფრო მეტად "ძვირი" ხდება მსგავს მატრიცაზე ოპერაციები.

**აღმოჩენილი კრიტიკული ბაგი:** train და test სეტებზე ცალ-ცალკე `pd.get_dummies()` გამოყენებამ გამოიწვია სხვადასხვა სვეტების დამატება, რაც პრობლემური აღმოჩნდა `preprocess()`-ის გამოყენების დროს, რატომ?

შეიძლება ტესტ სეტში არ მოხვედრილა იმ თვისების ქვეკატეგორიები, რომლებიც ტრეინ სეტში იყო, ეს კი იმას ნიშნავს, რომ ერთ სეტში არსებობს მსგავსი ქვეკატეგორიის სვეტი, მეორეში კი არა.

`x_train, x_test = x_train.align(x_test, join='left', axis=1, fill_value=0)` — თავდაპირველ `preprocess()` ფუნქციაში გამოყენებული ეს კოდი, უზრუნველყოფს ორივე სეტისთვის ერთნაირი სვეტების ქონას, თუ `x_test` არ შეიცავს `x_train`-ში არსებულ სვეტს, მაშინ მას ემატება 0-ებით სავსე სვეტი, რაც ხმაურს მატებს.

**კონკრეტული მაგალითი:** `RoofMatl` ცვლადში train-ში არსებობდა `ClyTile` კატეგორია, რომელიც გახდა baseline, ამიტომ `RoofMatl_CompShg` სვეტში CompShg-ის მქონე სახლებს ჰქონდათ მნიშვნელობა 1. test-ში `ClyTile` არ არსებობდა, ამიტომ `CompShg` თავად გახდა baseline — და ყველა CompShg სახლს მიენიჭა 0. შედეგად, ერთი და იგივე სახლი train-ში იყო 1, test-ში კი 0. [ამან კი გამოიწვია ძალიან დიდი ცდომილება ტესტ სეტზე](https://dagshub.com/ldavi22/cs229-231.mlflow/#/experiments/1/runs/0b363a1733944c35b2dde7eb41446b7b).

**გამოსწორება:** ტრეინ და ტესტ სეტის გაერთიანება, კოდირება, შემდეგ კი გახლიჩვა:

```python
combined = pd.concat([x_train.assign(_set='train'), x_test.assign(_set='test')])
combined = pd.get_dummies(combined, columns=nominal_cols, drop_first=True, dtype=int)
x_train = combined[combined['_set'] == 'train'].drop('_set', axis=1)
x_test = combined[combined['_set'] == 'test'].drop('_set', axis=1)
```

ამ შემთხვევაში ორივე ერთი და იგივე baseline-ს იყენებს და არ გვაქვს მსგავსი პრობლემა.

## NaN მნიშვნელობების დამუშავება

ამ დატასეტში NaN მნიშვნელობები სამ კატეგორიად იყოფა და თითოეული განსხვავებულ მიდგომას მოითხოვს.

### 1. NA ნიშნავს, რომ ცვლადი არ არსებობს

**მიდგომა:** თავდაპირველად ეს ცვლადები წავშალე რადგან ძალიან ბევრი NaN იყო. მაგრამ გადახედვის შემდეგ გამოჩნდა, რომ NA ინფორმაციის მატარებელი იყო და არა უბრალოდ შეუვსებელი ინფორმაცია. მაგალითად, თუ 95% სახლს `PoolQC = NA` აქვს, ეს არ ნიშნავს, რომ მონაცემების 95% აკლია, არამედ იმას რომ 95% სახლს აუზი არ აქვს.

ყოველი ჯგუფისთვის, NaN-ის მნიშვნელობის დასადასტურებლად გამოყენებული იქნა მასთან დაკავშირებული რიცხვითი ცვლადი:

| თვისებების ჯგუფი | შემოწმების გზა | შევსების გზა |
|---|---|---|
| Basement (BsmtQual, BsmtCond, BsmtExposure, BsmtFinType1/2) | `TotalBsmtSF == 0` ადასტურებს, რომ სარდაფი არ არის | 'None' → კოდირდება 0-ით |
| Garage (GarageType, GarageFinish, GarageQual, GarageCond) | `GarageArea == 0` ადასტურებს, რომ ავტოფარეხი არ არის | 'None' → კოდირდება 0-ით |
| FireplaceQu | `Fireplaces == 0` ადასტურებს, რომ ბუხარი არ არის | 'None' → კოდირდება 0-ით |

## Feature Selection

### გამოყენებული მიდგომები

**იდეა:** feature selection-ის მიზანია გაიფილტროს ისეთი თვისებები, რომელთა წვლილიც მოდელის ეფექტურობაში მინიმალურია, ამას გარდა შეიძლება თვისებები ერთმანეთის მიმართ მეტ-ნაკლებად წრფივად იყვნენ დამოკიდებულები, ეს კი იმას ნიშნავს, რომ საერთო ინფორმაციის მატარებლები არიან, რაც ხმაურს აჩენს მონაცემებში. ამ დავალებაში გამოყენებულია სამი მიდგომა:

### კორელაციით ფილტრაცია

ცვლადთა წყვილებს შორის ვითვლით კორელაციას. თუ ორ ცვლადს შორის კორელაცია threshold-ს (0.7) აღემატება, ნაკლებად სასარგებლო ცვლადი (target-თან ნაკლები კორელაციით) იშლება.

**იდეა:** კორელაცია ზომავს რამდენად წრფივად არის დამოკიდებული ორი ცვლადი ერთმანეთისადმი. თუ ორი ცვლადი ერთსა და იმავე წრფივ სიგნალს ატარებს, ორივეს შენარჩუნება ხმაურს ამატებს წრფივი მოდელებისთვის და აფუჭებს ლოგიკას ხეების გაყოფისთვის.

**პრობლემა:** მხოლოდ წრფივ კავშირებს ამჟღავნებს. ცვლადი, რომელსაც ძლიერი არაწრფივი კავშირი აქვს target-თან, შეიძლება არასწორად შეფასდეს.

### RFE (Recursive Feature Elimination)

RFE-ის გამოყენებით გაიფილტრა თვისებები და დარჩა მხოლოდ 100 საუკეთესო. ყოველ იტერაციაზე მოდელი იწვრთნება, ყველაზე მცირე აბსოლუტური კოეფიციენტის მქონე ცვლადი იშლება და პროცესი მეორდება.

### SHAP-ზე დაფუძნებული შერჩევა

SHAP მნიშვნელობები გამოითვალა საუკეთესო Random Forest მოდელისთვის.

SHAP პასუხობს კითხვას: "ყოველი ცვლადის წვლილი რა იყო ამ კონკრეტულ პროგნოზში?" და არა "რომელი ცვლადებია მნიშვნელოვანი ზოგადად."

**იდეა:** SHAP არის ყველაზე პრინციპული შერჩევის მეთოდი ხე-ზე დაფუძნებული მოდელებისთვის. კორელაციისგან განსხვავებით (მხოლოდ წრფივი) ან RFE-სგან (ერთ მოდელთან მიბმული), SHAP ასახავს მოდელის რეალურ გადაწყვეტილების პროცესს, მათ შორის არაწრფივ ეფექტებსა და ცვლადთა ინტერაქციებს.

## Training

### ტესტირებული მოდელები

#### წრფივი მოდელები

**OLS (Ordinary Least Squares):**

პირველი მოდელი, რომელიც გაიწვრთა უკვე გაფილტრულ დატაზე იყო, სტანდარტული წრფივი რეგრესია. არსებული მოდელებიდან ყველაზე დიდი RMSE ქონდა ამ მოდელს. იმისათვის, რომ ეფექტურობა გამეძლიერებინა feature selection პროცესის დროს გადავწყვიტე, რომ ჯერ კორელაციით ფილტრაცია მეცადა, შემდგომ კი RFE. ამ მიდგომამ ძალიან მინიმალური გაუმჯობესება მოგვცა, რეალურად გაუმჯობესდა cross-validation დროს არსებული train-validation სეტებზე ცდომილებები, ხოლო ტესტ სეტზე თითქმის იდენტური შედეგები ქონდათ. ეს იმას ნიშნავს, რომ მოდელს აღარ აქვს "განვითარების" შესაძლებლობა და მერყეობს 0.19-0.20 ცდომილების გარშემო. ამის გაუმჯობესება შეგვიძლია რეგულარიზაციით.

**წრფივი მოდელები რეგულარიზაციით (L1/L2):**

ვინაიდან ჩვენი მონაცემები შეიცავენ ბევრ სვეტებს (180>), რომელთა შორის ბევრი საერთო ინფორმაციას ატარებს, რეგულარიზაციის გარეშე, წრფივი მოდელი ცდილობს ყველას თანაბრად მოარგოს კოეფიციენტები, რაც რეალურად ხმაურზე მორგება არის და არა რეალურ მატარებელ სიგნალზე.

**Ridge (L2)** ამატებს ჯარიმას, რომელიც პროპორციულია კოეფიციენტების კვადრატების ჯამის. ყველა წონა ნულისკენ მიჰყავს, მაგრამ ზუსტად ნული არცერთი ხდება. ეს კი მოდელს აწონასწორებს, როდესაც ბევრი კორელირებული თვისება გვაქვს — იმის მაგივრად, რომ ერთ ცვლადს მიენიჭოს დიდი კოეფიციენტი, L2 თანაბრად ანაწილებს წონას მსგავს თვისებებზე.

**Lasso (L1)** ამატებს ჯარიმას, რომელიც პროპორციულია კოეფიციენტების მოდულის ჯამის. L2-ისგან განსხვავებით, შეუძლია კოეფიციენტების 0-ზე დაყვანა, რაც ეფექტურად თვისებების გადარჩევის ექვივალენტურია.

ამ სამი მოდელიდან ყველაზე კარგი შედეგი რეგულარიზირებულ მოდელებს ჰქონდათ, რაც მოსალოდნელი იყო (188 თვისება).

#### ხე-ზე დაფუძნებული მოდელები

ხე-ზე დაფუძნებული მოდელები **არ საჭიროებენ scaling-ს** — ეს არის მნიშვნელოვანი განსხვავება წრფივი მოდელებისგან. ხე მხოლოდ კითხვას სვამს "მეტია თუ ნაკლები X-ზე?" — რიცხვების მასშტაბს არავითარი მნიშვნელობა არ აქვს.

| მოდელი | Feature Selection | ოპტიმიზაციის მეთოდი |
|--------|-------------------|---------------------|
| Decision Tree | ყველა ცვლადი, კორელაციით გაფილტრული | GridSearchCV |
| Random Forest | ყველა ცვლადი, SHAP-ით და კორელაციით გაფილტრული | RandomizedSearchCV (100 იტერაცია) |
| XGBoost | ყველა ცვლადი | Optuna (100 trial) 5-fold CV + early stopping |

ამ სამ მოდელში ყველაზე კარგად იმუშავა Random Forest-მა, რაც ერთი შეხედვით არ იყო მოსალოდნელი, ვინაიდან XGBoost-იც იყო გამოყენებული, მაგრამ ამ ყველაფრის მიზეზი არის ის რომ, XGBoost ძალიან ბევრი პარამეტრი აქვს და მისი ოპტიმიზაცია, ისე რომ, ოვერფიტი არ მოხდეს და თან ცდომილებაც პატარა იყოს, ბევრ ტრენინგს მოითხოვს. ჩემი გაშვებული cross-validation-ის შემდეგ თითქმის არც-ერთ ვერსიას ცუდი შედეგი არ ჰქონია, პრობლემა ის იყო, რომ ტრენინგ სეტზე ძალიან დაბალი ერორი იყო, თითქმის 0, რეალურად იზეპირებდა ინფორმაციას, მაგრამ ამის მიუხედავად ტესტ სეტზე მაინც კარგ შედეგს დებდა. ვინაიდან ტრენინგ სეტს იზეპირებდა, მაგრამ თან კარგ შედეგს დებდა, ეს მოდელი ჩავთვალე, რომ არ მუშაობდა კარგად.

Random Forest მუშაობს ყველაზე ოპტიმალურად, ტესტ სეტზე ერორი [0.14, 0.15] მერყეობს, რაც ძალიან კარგია. მასზე ასევე გაიტესტა სხვადასხვა feature selection მეთოდები, მაგრამ შედეგი იგივე ქონდა, აქედან კი შეიძლება დავასკვნათ, ის რომ, ზოგი მიდგომა მოდელისთვის სპეციფიურია და არ შველის სხვა ტიპის მოდელებს.

ამ სამში ყველაზე ცუდად Decision Tree-მ იმუშავა, რაც მოსალოდნელი იყო, ყველაზე სუსტია ამათთან შედარებით, მიუხედავად ამისა მისი ცდომილება მაინც [0.19, 0.20] მერყეობს, რაც არც ისე ცუდია.

### Hyperparameter ოპტიმიზაციის მიდგომა

**GridSearchCV** — გამოვიყენე თითქმის ყველა მოდელის ტრენინგის დროს. ეს კონკრეტული მეთოდი იძლევა საშუალებას, რომ ყველა შესაძლო ჰიპერპარამეტრის ვარიანტი გაიტესტოს კროს-ვალიდაციის დროს. მძიმე ხის მოდელებისთვის ამ მეთოდს არ ვიყენებდი, ვინაიდან ძალიან ბევრი ვარიანტი იყო და ძალიან ბევრი დრო სჭირდებოდა. მათვის ვიყენებდი RandomizedSearchCV-ს.

**RandomizedSearchCV** — შემდეგი მეთოდი ეფექტური იყო XGBoost-ის და RandomForrest-ის დროს. განსხვავებით GridSearch-ისგან ეს მეთოდი ყველა შესაძლო ვარიანტს არ განიხილავს, არამედ მხოლოდ შემთხვევითად შერჩეული პარამეტრების კომბინაციას.

წრფივი მოდელებისთვის GridSearch-ში Scaler-ის შერჩევაც შედიოდა (StandardScaler, MinMaxScaler, RobustScaler). **რატომ სჭირდება წრფივ მოდელს სკალირება?** წრფივი მოდელი თითოეული თვისებისთვის კოეფიციენტს სწავლობს. თუ GrLivArea 500-5000 დიაპაზონშია, ხოლო OverallQual 1-10-ში, კოეფიციენტები სრულიად განსხვავებული მასშტაბის ხდება და ვინაიდან წრფივი მოდელები Gradient Descent იყენებენ ოპტიმიზაციისთვის, მასშტაბი დიდ როლს თამაშობს ამ პროცესის დროს.

**Optuna** — ვინაიდან GridSearch/RandomizedSearchCV არ აქვს early stopping-ის საპორტი, რაც გულისხმობს იმას, რომ თუ მოდელი აღარ უმჯობესდება ახალი ხეების დამატებით, მაშინ გაჩერდეს ისეთ იტერაციაზე სადაც ყველაზე ნაკლები ხე (estimator) იყო გამოყენებული. ეს თვისება XGBoost-ის შემთხვევაში ძალიან მნიშვნელოვანი იყო და ვინაიდან Optuna-ს ქონდა ამის საშუალება გამოვიყენე ოპტუნა. ერთი მინუსი აქვს Optuna-ს — კროს-ვალიდაცია ხელით უნდა გაიწეროს, თუ წინა მეთოდებში KFold შეგვეძლო, რომ გადაგვეწოდებინა პარამეტრებში, აქ მსგავს რამეს ვერ ვიზამთ.

### საბოლოო მოდელის შერჩევის დასაბუთება

ვინაიდან ყველაზე კარგად Lasso Regression მუშაობს, ამ მოდელის მიხედვით დავაპროგნოზე საბოლოო შედეგი. **Public Score: 0.1428**

## MLflow Tracking

### MLflow ექსპერიმენტების ბმული

ყველა ექსპერიმენტი ფიქსირდება: [https://dagshub.com/ldavi22/cs229-231.mlflow](https://dagshub.com/ldavi22/cs229-231.mlflow)

### ჩაწერილი მეტრიკების აღწერა

ყოველი ექსპერიმენტის run-ისთვის შემდეგი მეტრიკები ჩაიწერება:

| მეტრიკა | აღწერა |
|---------|--------|
| `train_rmse` | საშუალო RMSE სატრენინგო fold-ებზე cross-validation-ის დროს |
| `val_rmse` | საშუალო RMSE სავალიდაციო fold-ებზე (ძირითადი შედარების მეტრიკა) |
| `val_std` | validation RMSE-ის სტანდარტული გადახრა fold-ებს შორის |
| `test_rmse` | RMSE გამოყოფილ test ნაკრებზე |
| `overfit_gap` | `val_rmse - train_rmse` — ზომავს, რამდენად უარესად მუშაობს მოდელი ახალ მონაცემებზე |

**ინტერპრეტაცია:**
- `val_rmse` გამოიყენება ექსპერიმენტების შესადარებლად და საუკეთესო მიდგომის შესარჩევად
- `test_rmse` არის საბოლოო რეალობის შემოწმება — თუ ის val_rmse-ზე ბევრად უარესია, მოდელი CV fold-ებისთვის ზედმეტად ოპტიმიზირდა
- `overfit_gap` ახლოს 0-თან ნიშნავს, რომ მოდელი კარგად ზოგადდება; დიდი სხვაობა overfitting-ზე მიუთითებს

### ჩაწერილი პარამეტრები

თითოეული run ასევე აფიქსირებს: მოდელის სახელს, საუკეთესო ჰიპერპარამეტრებს (grid/random/optuna ძიებიდან), ცვლადების რაოდენობას, სატრენინგო ნიმუშების რაოდენობას, feature selection მეთოდს და მეთოდის სპეციფიკურ პარამეტრებს.
